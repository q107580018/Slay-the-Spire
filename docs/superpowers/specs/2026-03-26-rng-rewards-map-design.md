# RNG, Rewards, And Map Distribution Design

Date: 2026-03-26
Status: Draft for review

## 1. 背景

当前项目已经具备单局 run 的基础闭环，但“肉鸽感”仍然不够强，核心问题不在战斗主循环，而在随机系统还没有形成接近原版《Slay the Spire》的体验：

- 地图拓扑已经由 seed 驱动随机生成，但房型分配仍偏“原型级均匀随机”
- 问号房已经会从事件池抽取事件，但事件出现规则仍然偏薄
- 战斗奖励目前还不是接近原版的“金币 + 掉落物 + 3 张卡牌候选”
- 普通敌人、精英、Boss 的遭遇抽样仍缺少权重控制

用户希望本轮直接往“高拟真复刻”靠，但项目当前仍是：

- 单角色：`ironclad`
- 单幕：`act1`
- 终端编号菜单驱动
- 内容量有限，无法一口气完整复刻原版全部 encounter table、卡池与事件池

因此，本轮设计目标是：

- 在不推翻现有架构的前提下，把随机系统升级到“原版风格且可持续扩展”
- 优先解决玩家最强感知的三类随机性：
  - 地图房型分布
  - 遭遇与事件抽取
  - 战后奖励生成

## 2. 目标

- 建立统一、稳定、可复现的随机派生规则
- 将地图房型分配从“允许集合均匀抽”升级为“楼层分段权重 + 约束修正”
- 将普通战、精英战、Boss、事件改为内容驱动的加权抽样
- 将普通战和精英战奖励升级为接近原版的结构：
  - 金币随机区间
  - 药水掉落概率
  - 3 张卡牌候选选 1 张
  - 预留“跳过”
- 保持存档恢复后的随机结果稳定，不因菜单浏览或读取顺序变化而漂移
- 为后续继续补原版规则预留扩展点，而不是做一次性硬编码

## 3. 非目标

- 不在本轮完整复刻原版所有隐藏随机表与底层 RNG 推进细节
- 不在本轮扩展多角色、多幕、多 Boss 规则差异
- 不在本轮完整复刻原版全部事件条件、一次性事件、稀有事件分类
- 不在本轮一次性补齐原版完整铁甲奖励卡池
- 不顺手重写整个战斗系统、菜单系统或存档系统

## 4. 设计选择

本次采用：

- **配置驱动的高拟真随机系统**

不采用：

- 各模块继续各自手写随机逻辑
- 为了“像原版”而引入一整套难以维护的全局 RNG 指针推进模型
- 在地图、事件、奖励里塞大量角色或 Act 专属硬编码分支

原因：

- 现有代码已经具备 `seed + room_id/category` 派生随机的雏形，适合继续系统化
- 配置驱动更符合当前项目内容 JSON 的组织方式
- 高拟真体验的关键是“分布和约束像原版”，不一定需要复制原版内部实现细节
- 稳定派生 RNG 比“全局指针前进”更容易测试，也更利于存读档恢复

## 5. 范围

### In Scope

- 统一 RNG 派生规则
- 地图房型分配规则升级
- 遭遇池与事件池加权抽样
- 普通战 / 精英战 / Boss 奖励生成规则升级
- 奖励菜单支持 3 张卡牌候选的领取与跳过
- 为卡牌奖励引入独立奖励池配置
- 同步内容 JSON 到仓库根目录 `content/` 和打包目录 `src/slay_the_spire/data/content/`
- 补齐领域测试、用例测试和至少 1 条 E2E 冒烟路径

### Out of Scope

- 原版完整事件排他规则与所有 once-per-run 事件
- 原版完整稀有卡、非普通卡、颜色无关卡、诅咒卡掉落逻辑
- 原版完整药水稀有度和遗物等级体系
- 原版多怪战、遭遇编组、战斗阶段权重轮换
- 原版完整地图视觉布局算法

## 6. 随机系统设计

### 6.1 核心原则

所有随机结果都必须满足：

- 同一 `seed` 下可复现
- 同一房间的结果不会因其他界面操作而变化
- 存档恢复后结果保持一致
- 不同随机类别彼此隔离

### 6.2 随机流

定义 4 条主随机流：

- `map`
- `encounter`
- `event`
- `reward`

具体派生时再细分子类别，例如：

- `map:topology`
- `map:room_types`
- `encounter:combat`
- `encounter:elite`
- `encounter:boss`
- `event:room`
- `reward:gold`
- `reward:potion`
- `reward:card`

推荐统一提供 helper，例如：

- `rng_for_run(seed, category)`
- `rng_for_room(seed, room_id, category)`

当前散落在 `map_generator.py`、`enter_room.py`、`reward_generator.py` 的 `Random(...)` 构造应逐步收拢到统一入口。

### 6.3 取舍

本轮不模拟“原版全局 RNG 状态机”。  
改用“`seed + scope + category` 派生”的稳定随机。

这不是原版的字节级实现，但对本项目更合理：

- 玩家感知结果足够像原版
- 测试更稳定
- 存档恢复不容易出错
- 未来继续精细化时仍可替换底层 helper

## 7. 地图分布设计

### 7.1 保留部分

保留现有：

- 分层拓扑生成
- 分支连线生成
- Boss 固定在最终层

这些已经足够支撑终端原型，不需要重写地图骨架。

### 7.2 升级部分

当前房型分配逻辑是：

- 根据楼层段得到允许房型集合
- 从允许集合中均匀抽取

这会导致结果虽然“随机”，但节奏不像原版。  
本轮改为：

- 楼层段权重抽样
- 最早出现楼层限制
- 邻接/连续约束
- 全图最低数量保证

### 7.3 建议约束

对 `act1` 增加接近原版风格的控制：

- 前几层不出现精英
- 商店与休息不会过早出现
- 前段以普通战和问号房为主
- 中段稳定引入商店与休息
- 后段提高精英出现权重
- 同一路径上尽量避免连续出现过多相同特殊房

### 7.4 配置化方式

在 `act1_map.json` 的 `map_config` 中扩展字段，例如：

- `room_type_weights`
- `min_floor_by_room_type`
- `min_count_by_room_type`
- `max_consecutive_special_room_types`

不要求命名必须完全如此，但语义应覆盖：

- 各楼层段房型权重
- 每类房型最早楼层
- 全图最低数量
- 连续/邻接约束

### 7.5 约束处理策略

推荐处理顺序：

1. 先按拓扑生成节点
2. 先给每个节点按楼层权重抽候选房型
3. 再做连续约束修正
4. 最后做全局最低数量兜底修正

这样可保持实现清晰，也方便测试每一层规则。

## 8. 敌人与事件抽样设计

### 8.1 统一抽样模型

普通战、精英、Boss、事件统一采用：

- 内容池成员
- 成员权重
- 稳定加权抽样

禁止继续在这些路径上使用简单的 `rng.choice(ids)`。

### 8.2 敌人池

现有敌人池 JSON 主要是成员列表。  
本轮升级为成员可带权重，例如概念上支持：

- `enemy_id`
- `weight`

普通战、精英战、Boss 都适用。

如果后续仍保持“一个文件就是一个池”的结构，则池文件中的每个条目需要支持权重字段。内容加载器与 registry 需同步支持。

### 8.3 事件池

事件池同样改为成员可带权重。

第一轮建议支持：

- `event_id`
- `weight`
- 可选 `once_per_run`

`once_per_run` 是对原版体验最有价值、同时复杂度可控的增强。  
如果实现时发现需要改动存档结构，则应通过 `run_state` 或 `room_state.payload` 留下“已见事件集合”的兼容扩展位。

### 8.4 未来预留

本轮不一定全部实现，但设计上预留后续扩展：

- 基于当前金币/HP/遗物的事件过滤
- 同一局近期遭遇去重
- 按战斗阶段切换普通战 encounter table

## 9. 奖励系统设计

### 9.1 普通战奖励

普通战胜利后生成：

- 金币奖励：在配置区间内随机
- 药水掉落：按概率决定是否掉落
- 卡牌奖励：3 张候选，玩家选 1 张
- 跳过选项：允许不拿卡

当前实现中的“固定金币 + 固定 1 张卡”应废弃。

### 9.2 精英战奖励

精英战胜利后生成：

- 更高的金币奖励区间
- 保底遗物
- 3 张卡牌候选
- 药水掉落逻辑可复用统一规则，必要时允许更高概率

### 9.3 Boss 战奖励

Boss 战继续保留：

- 高额金币
- Boss relic 三选一

是否在本轮同时补 Boss 卡牌奖励，不强制要求；以不破坏现有 Boss 奖励流程为优先。

### 9.4 卡牌奖励池

卡牌奖励不能再从当前固定常量中生成。  
需要为角色引入独立的奖励卡池配置，至少区分：

- starter deck 中的基础卡
- 可出现在普通战/精英战奖励中的卡
- 可出现在商店中的卡

最小方案：

- 在现有角色卡内容中新增“是否可出现在战斗奖励”字段

更推荐的方案：

- 为角色或卡内容增加明确的 pool/tag 语义，例如 `reward_pool`

### 9.5 当前内容量问题

当前奖励卡内容过少，若直接实现 3 选 1，会导致重复度很高、体验接近伪随机。  
因此本轮除了系统改造，还应补一批最小可玩的 `ironclad` 奖励卡内容，使“三选一”至少具有基本选择性。

这部分不要求一次性补齐原版卡池，但要满足：

- 候选数量足以经常抽出 3 张不同卡
- 攻防功能有基本区分
- 不只是在 `strike / defend / bash` 上打转

## 10. 数据模型与内容改动

### 10.1 内容文件

预期会修改：

- `content/acts/act1_map.json`
- `src/slay_the_spire/data/content/acts/act1_map.json`
- `content/enemies/act1_basic.json`
- `src/slay_the_spire/data/content/enemies/act1_basic.json`
- `content/enemies/act1_elites.json`
- `src/slay_the_spire/data/content/enemies/act1_elites.json`
- `content/enemies/act1_bosses.json`
- `src/slay_the_spire/data/content/enemies/act1_bosses.json`
- `content/events/act1_events.json`
- `src/slay_the_spire/data/content/events/act1_events.json`
- 角色相关卡牌内容文件及其打包副本

### 10.2 内容读取

`catalog.py` / registry / provider 需要支持：

- 池成员权重读取
- 事件的可选 once-per-run 元数据
- 奖励卡池相关字段

### 10.3 存档兼容

若引入 once-per-run 事件或更显式的奖励状态，需要审视：

- `RunState`
- `RoomState`
- `save_game.py`
- `load_game.py`
- 对应序列化测试

要求：

- 不破坏已有 JSON 兼容性
- 新字段有默认值

## 11. 用例与终端交互改动

### 11.1 `enter_room`

需要改造：

- 普通战 / 精英 / Boss 改为加权抽样
- 事件房改为加权抽样
- 如启用 once-per-run 事件，需要在此处过滤候选

### 11.2 奖励生成

`reward_generator.py` 需要从当前简化实现升级为：

- 按房间类型区分奖励表
- 金币区间随机
- 药水掉落判定
- 3 张卡牌候选无放回抽样
- 精英遗物 / Boss relic 逻辑保持清晰分层

### 11.3 菜单与渲染

终端菜单需要支持：

- 显示 3 张卡牌候选
- 明确区分金币、药水、遗物、卡牌奖励
- 提供“跳过卡牌”操作

若现有奖励菜单假设“一次只有 1 张卡”，则需要同步更新：

- app 菜单定义
- terminal renderer / inspect
- 对应测试

## 12. 测试策略

### 12.1 地图生成测试

新增或调整测试以验证：

- 同一 seed 下地图结果稳定
- 不同 seed 下地图存在明显差异
- 房型满足最早楼层限制
- 全图满足最低数量保证
- 连续约束生效

### 12.2 遭遇与事件测试

验证：

- 加权抽样对固定 seed 结果稳定
- 权重为 0 或缺失时行为符合预期
- 事件池为空、过滤后为空时有明确错误
- once-per-run 事件不会重复出现

### 12.3 奖励测试

验证：

- 普通战奖励包含金币与 3 张卡牌候选
- 精英战奖励包含遗物与 3 张卡牌候选
- 药水掉落逻辑可复现
- 同一次奖励生成不会出现重复卡牌候选
- “跳过卡牌”不会修改牌组
- 选择卡牌会向牌组写入真实实例

### 12.4 E2E 冒烟

至少覆盖一条完整路径：

- 开局
- 进入普通战
- 战斗胜利
- 看到 3 张卡牌候选
- 选择其中 1 张
- 奖励正确写回 `run_state`

如果精英奖励菜单与普通奖励菜单共用，建议再补一条精英路径断言。

## 13. 风险与取舍

### 13.1 内容不足

最大风险不是代码，而是内容量。  
如果不补最小奖励卡池，系统虽然变成“三选一”，体验仍不会像原版。

### 13.2 过早高保真

如果一次性引入：

- 稀有度
- 多层奖励池
- 多阶段遭遇表
- 复杂一次性事件约束

则会显著拉高实现成本，并增加存档与测试负担。

因此建议先完成“高拟真骨架”，再逐轮细化。

### 13.3 存档与菜单回归

奖励结构从单卡变成多候选后，最容易回归的地方是：

- 奖励菜单
- 领取逻辑
- 重载存档后的奖励恢复

这些路径必须靠测试兜住。

## 14. 实施建议

推荐实施顺序：

1. 统一 RNG helper
2. 升级池内容模型，支持权重
3. 改 `enter_room` 的遭遇与事件抽样
4. 改地图房型分配
5. 改奖励生成与奖励菜单
6. 补最小奖励卡内容
7. 补测试与 E2E

这样可以把“随机入口”先稳定，再往上接地图和奖励，降低调试成本。

## 15. 预期结果

完成后，项目应具备以下可感知变化：

- 同一个 seed 的 run 可复现，但不同 seed 的地图节奏差异明显
- 问号房不再只是“随机有事件”，而是像原版那样成为更重要的不确定性来源
- 普通战和精英战的遭遇出现频率不再显得机械平均
- 战斗胜利后能真正看到“3 张卡牌里挑 1 张”的肉鸽奖励体验
- 整体随机性更像《Slay the Spire》，同时仍然保持当前终端原型架构的可维护性
