# Lagavulin Core Behavior Design

Date: 2026-03-29
Status: Draft for review

## 1. 背景

当前项目中的 `lagavulin` 只实现了“睡眠 3 次敌方行动后，固定打 18 伤”的简化版本：

- 没有原版核心的 `Siphon Soul`
- 没有对玩家 `strength` / `dexterity` 的负向状态影响
- 敌人意图、效果摘要和战斗日志也无法完整表达“减力 / 减敏”这类行为

用户本轮要求不是做“看起来像 Lagavulin 的占位怪”，而是补到更接近原版《Slay the Spire》1 代的核心战斗行为。

## 2. 目标

- 让 `lagavulin` 具备更接近原版的核心行动模式
- 补齐 `Siphon Soul` 对玩家的减力、减敏效果
- 让战斗计算真实受负向 `strength` / `dexterity` 影响
- 让意图预览、效果摘要、战斗日志能正确展示这些变化
- 保持当前项目的内容驱动和通用效果架构，不为 `lagavulin` 增加硬编码特例

## 3. 非目标

- 不在本轮精确复刻原版全部数值、AI 规则和隐藏判定
- 不在本轮实现“被攻击后提前苏醒”等更细的原版互动
- 不把整个状态系统重构为完整原版 Buff/Debuff 框架
- 不顺手扩展其他敌人的行为，只做支撑 `lagavulin` 所需的最小通用能力

## 4. 设计选择

### 4.1 采用通用负状态通路，不做敌人专用特例

采用：

- 让通用效果层支持 `strength` / `dexterity` 的正负变化
- 让 `lagavulin` 的内容定义直接通过 `effects` 描述 `Siphon Soul`

不采用：

- 在战斗流程里写死“如果敌人是 lagavulin，则额外减玩家力量和敏捷”

原因：

- 当前项目已经以 JSON 内容定义敌人行为，继续沿用数据驱动更一致
- 若把减力减敏写成 `lagavulin` 特判，后续再做类似敌人或事件时还会继续污染核心战斗流
- 负向 `strength` / `dexterity` 是具有复用价值的通用能力，不应只服务一个精英怪

### 4.2 只补原版核心行为，不追求一次性“完美复刻”

采用：

- 睡眠阶段
- 苏醒后攻击
- `Siphon Soul`
- 再次攻击并循环

不采用：

- 一次性补全所有原版唤醒细节、数值浮动、复杂 AI 约束

原因：

- 当前仓库的目标是可玩原型，不是逐帧复刻
- 用户明确接受“原版核心战斗行为”范围
- 先补关键行为闭环，能以最小改动显著提升还原度

## 5. 总体方案

### 5.1 `lagavulin` 行动序列

将 `lagavulin` 的内容定义调整为：

- 开场 `sleep` 3 次敌方行动
- 苏醒后执行一次重击
- 执行 `Siphon Soul`
- 再执行一次重击
- 之后在攻击与 `Siphon Soul` 之间循环

本轮不强求逐项对齐原版内部 move 名称，只要求玩家可感知行为与原版核心一致。

### 5.2 通用状态语义

#### `strength`

- 已存在的正向力量逻辑保留
- 新增负向力量支持
- 伤害结算读取实体当前 `strength` 总值，允许为负
- 最终伤害最低为 `0`，不允许出现负伤害

#### `dexterity`

- 新增运行时有效的 `dexterity`
- 格挡结算读取实体当前 `dexterity` 总值，允许为负
- 最终获得格挡最低为 `0`
- 本轮不扩展到所有未来可能的格挡来源语义差异，只覆盖当前统一的 `block` 效果结算

### 5.3 `Siphon Soul` 的内容表达

`lagavulin` 的 `Siphon Soul` 直接在 `move_table` 中描述为两个效果：

- 对玩家施加负向 `strength`
- 对玩家施加负向 `dexterity`

这样敌人定义本身就能表达行为，不需要在战斗流程里新增 `lagavulin` 专属动作类型。

## 6. 具体改动点

### 6.1 内容层

同步修改以下两份内容文件中的 `lagavulin`：

- `content/enemies/act1_elites.json`
- `src/slay_the_spire/data/content/enemies/act1_elites.json`

调整项：

- `move_table`
- 中文名称保持现状
- 维持现有内容结构，不新增只被单个敌人使用的 schema 字段

### 6.2 领域层

需要调整：

- `effect_resolver`
- `effect_types` 中与状态效果相关的常量
- 支撑状态结算的少量辅助函数

行为要求：

- `strength` 效果允许正负数
- 新增 `dexterity` 效果类型
- `damage` 结算时应用 `strength`
- `block` 结算时应用 `dexterity`
- 应用后状态可保留负层数

### 6.3 展示与日志层

需要调整：

- 效果摘要
- 敌人意图预览
- 战斗事件生成
- 战斗日志文案

展示要求：

- 正向状态继续显示“获得 X 力量 / 敏捷”
- 负向状态显示“失去 X 力量 / 敏捷”或等价中文
- `lagavulin` 的意图能明确体现 `Siphon Soul` 会减力减敏，而不是只显示抽象 move 名

## 7. 数据流

1. 进入精英战时，`lagavulin` 仍带有 `sleeping` 状态
2. 睡眠期间，敌人预览与日志显示沉睡
3. 睡眠结束后，按 `move_table` 进入攻击和 `Siphon Soul` 循环
4. `Siphon Soul` 通过通用 effect queue 产生负向 `strength` / `dexterity`
5. `effect_resolver` 将这些状态写入玩家状态列表
6. 后续玩家出牌时：
   - 伤害效果读取负向 `strength`
   - 格挡效果读取负向 `dexterity`
7. Rich/Textual 共用的摘要与日志层显示正确中文描述

## 8. 风险与约束

### 8.1 负状态会影响现有正状态计算

当前系统默认 `strength` 只会增加伤害。本轮改为允许负值后，需要确认：

- 玩家已有正力量时，再吃减力，结果是净值相加
- 不会因为负值导致异常日志或负伤害

### 8.2 `dexterity` 是新增运行时能力

虽然 UI 已有“敏捷”标签，但当前运行时尚未真正使用它。补上后要注意：

- 格挡药水、防御牌、敌人加格挡等统一走 `block` 结算时都会受影响
- 最终不能出现负格挡或反向扣格挡

### 8.3 测试覆盖必须先于实现

本轮同时涉及：

- 内容定义
- 效果解析
- 战斗流程
- 日志展示

如果只改内容不补测试，很容易出现“意图显示正确，但实际不生效”或“实际生效，但日志错误”的回归。

## 9. 测试策略

先写失败测试，再实现。至少覆盖：

- `lagavulin` 睡眠后会进入包含 `Siphon Soul` 的循环
- `Siphon Soul` 会对玩家施加负向 `strength` 和 `dexterity`
- 玩家负力量会降低后续伤害
- 玩家负敏捷会降低后续格挡，但结果不低于 `0`
- 敌人意图预览会显示减力减敏
- 战斗日志会显示对应中文文案

## 10. 验收标准

- `lagavulin` 不再只是“睡三回合后一直打 18”
- 玩家能在战斗中真实受到减力减敏影响
- 相关 UI 文案与日志能清楚表达行为
- 根目录 `content/` 和打包目录 `src/slay_the_spire/data/content/` 内容保持同步
- 相关测试通过，且不破坏现有战斗主流程
