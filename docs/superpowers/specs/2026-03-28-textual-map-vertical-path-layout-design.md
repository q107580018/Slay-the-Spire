# Textual Map Vertical Path Layout Design

Date: 2026-03-28
Status: Draft for review

## 1. 背景

当前 `Textual` 左侧地图已经能展示整张 Act 地图，并支持滚动、点击、悬停和当前节点自动定位。但展示方式仍更接近“静态网格图”：

- 节点主要按 `row / col` 原始坐标排布
- 连线更像几何连接，而不是原版《Slay the Spire》的纵向路径感
- 分支多时，玩家需要脑补哪些节点属于同一条未来路线
- 从起点往下看时，难以直观看出“这一层的分支会继续通向哪里”

用户本轮要求明确：

- **只改左侧地图本体**
- **改成更像原版的纵向路径图**
- **每一层把后续分支直接摊开**
- **从起点开始就能看懂完整可行走路线**

因此，本轮不新增独立路线面板，不改右侧信息区，不重写地图生成规则。

## 2. 目标

- 将 `Textual` 左侧地图改造成“纵向路径图”而不是“坐标网格图”
- 让玩家从起点开始，就能顺着地图直接看出每一层后续分支的去向
- 保持当前节点、可达节点、悬停和点击交互
- 保持地图可滚动，并继续支持自动定位到当前节点附近
- 让连线尽量避免交叉，提升字符终端里的路线可读性

## 3. 非目标

- 不修改领域层地图生成逻辑
- 不修改 `ActState` / `ActNodeState` 的数据结构
- 不新增独立的“路线树面板”
- 不改右侧菜单、日志、战斗面板或奖励面板
- 不追求原版像素级美术复刻
- 不在本轮重写 Rich 侧的完整地图展示

## 4. 设计选择

本轮采用：

- **Textual 自绘地图重构为“布局层 + 路由层 + 节点层”**

不采用：

- 只给现有地图加更强高亮
- 直接照 `row / col` 画图不做布局重排
- 为每个节点单独做 Textual 子组件

原因：

- 用户要的是“地图本体更像原版”，核心问题在布局与连线路由
- 现有地图已经证明单个 `ScrollView` 可以承载整图，自绘继续最稳
- 先把布局算法独立出来，后续更容易调连线、间距和高亮策略

## 5. 总体方案

### 5.1 分层

Textual 地图渲染拆成三个逻辑层：

1. 布局层：为节点分配新的显示坐标
2. 路由层：为每条边生成终端可读的连线路径
3. 节点层：在连线之上覆盖节点图标、标签和状态高亮

这样可以避免“节点坐标”和“显示坐标”混在一起，后续调版面时只改布局层即可。

### 5.2 视觉目标

地图视觉上更接近原版的纵向爬塔路径：

- 纵向推进清晰
- 同层分支横向摊开
- 相邻层之间的连接关系一眼可见
- 分叉和合流尽量不互相干扰

本轮不强求完全复制原版的所有空白比例，但要达到“从起点往下能顺着路线读”的程度。

## 6. 布局设计

### 6.1 输入数据

仍然只使用现有数据：

- `ActState.nodes`
- `ActNodeState.row`
- `ActNodeState.col`
- `ActNodeState.next_node_ids`
- `ActState.current_node_id`
- `ActState.reachable_node_ids`

本轮不引入新的地图数据字段。

### 6.2 显示坐标

显示坐标不再直接复用原始 `col`，而是为每个节点计算独立的 `display_lane`：

- `row` 继续决定纵向层级
- `display_lane` 决定横向展开位置
- 同一层节点按“路线相关性”排序，而不是单纯按原始列号排序

推荐规则：

- 当前节点所在路线优先居中
- 子节点优先继承父节点的横向相对顺序
- 多父节点的合流点使用父节点横向位置的中位数或中心值
- 同一层的不同分支之间保留最小间距，避免节点挤压

这类布局本质上是“有层级约束的路径展开”，目标不是几何严格最优，而是可读性优先。

### 6.3 稳定性

显示坐标必须稳定：

- 同一张地图重复渲染结果一致
- 悬停、刷新、滚动不会改变节点左右顺序
- 自动定位后不会让整图因为重排而跳动

因此，布局规则必须只依赖地图结构和稳定排序，不依赖鼠标、刷新时机或视口大小。

## 7. 连线路由设计

### 7.1 路由原则

连线不再按“画到几何中点”的思路处理，而是按“通道”处理：

- 每一层之间预留固定的连接通道
- 边沿着预定通道从父节点流向子节点
- 优先使用短折线和竖向连接
- 尽量避免让两条路线在同一通道中交叉穿越

### 7.2 字符表达

终端里使用可组合的 Unicode 线条字符，例如：

- 竖线：`│`
- 左下 / 右下斜线：`╱` / `╲`
- 分叉和汇合：`├` `┤` `┬` `┴` `┼`

原则是：

- 单一路线用简单字符
- 分叉点和合流点用 junction 字符
- 节点覆盖在连线之上

### 7.3 画线顺序

绘制顺序固定为：

1. 先写入背景和连线
2. 再写入节点图标和标签
3. 最后写入当前节点、可达节点、悬停态样式

这样可以避免节点文字被线条切碎。

## 8. 节点设计

### 8.1 节点内容

节点视觉继续保留“图标 + 标签”，但体积需要比现有地图更轻：

- 战斗：`⚔`
- 精英：`💀`
- Boss：`👑`
- 事件：`❓`
- 商店：`🛒`
- 休息：`🔥`

标签维持中文，便于终端阅读。

### 8.2 命中区域

节点在视觉上变小后，点击和悬停命中区域仍然需要覆盖完整节点体积，并允许略大于可见字符范围，以免鼠标交互变得难用。

### 8.3 状态表达

保留现有状态语义：

- 当前节点：最强高亮
- 可达节点：次级高亮
- 普通节点：类型色彩

本轮不引入新的“路线进度”状态变量，路线信息通过布局和连线来表达。

## 9. 交互与视口

### 9.1 点击

可达节点仍可点击进入，不可达节点仍只允许浏览。

### 9.2 悬停

悬停仍只做轻量反馈，不打断地图主体可读性。

### 9.3 自动定位

保持当前节点自动定位，但新的布局可能更宽，因此：

- 视口定位仍以当前节点为锚点
- 进入新节点后重新定位
- 横向偏移优先保证当前节点可见，纵向偏移优先给上方未来路线留阅读空间

## 10. 实施范围

本轮主要修改：

- `src/slay_the_spire/adapters/textual/map_widget.py`
- `src/slay_the_spire/adapters/textual/slay_app.py`
- `tests/adapters/textual/test_slay_app.py`

如需抽出公共帮助函数，可新增一个 Textual 适配层内的局部模块，但不扩展到领域层。

## 11. 测试策略

需要补的测试重点如下：

- 地图在典型分叉图上能稳定生成，不出现节点重叠
- 连线在分叉和合流时保持可读，不会把整图画乱
- 当前节点和可达节点仍能正确高亮
- 自动定位仍然把当前节点带入视口
- 悬停和点击命中逻辑不因新布局失效

建议至少补以下类型：

- 布局单测
- Textual 地图渲染单测
- 当前节点视口定位测试

## 12. 风险与约束

### 12.1 宽度膨胀

纵向路径图往往会比现有规则网格更宽。这个风险可以接受，但需要保留横向滚动，并避免节点间距过度压缩。

### 12.2 交叉控制

如果布局只靠简单偏移，地图会在分支多时再次变乱。因此布局层必须比“给每个节点加一点 wobble”更强，至少要有稳定的 lane 分配。

### 12.3 双端一致性

如果只改 Textual 不改其它地方，用户仍可能在别的页面看到旧地图风格。当前本轮只要求左侧地图，因此这不算问题，但需要避免不小心波及 Rich 侧的非目标逻辑。

## 13. 验收标准

- 左侧 Textual 地图看起来是纵向路径图，而不是纯坐标网格图
- 从起点开始，玩家可以顺着图直接读出每层后续分支
- 连线交叉显著减少，且不会让节点关系难以辨认
- 当前节点与可达节点的高亮仍然明确
- 现有点击、悬停、滚动、自动定位继续可用
