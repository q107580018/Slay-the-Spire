# Terminal Rich UI Design

## 背景

当前项目已经具备最小可玩的中文终端骨架，但界面仍然属于“逐行文本 + 数字菜单”的日志式输出。用户希望界面更接近 [`sts2-cli`](https://github.com/wuhao21/sts2-cli) 的观感，重点提升：

- 血条与状态的视觉化表达
- 面板式分区布局
- 一屏一个决策点的完整上下文
- 纯中文显示与数字输入并存

本次工作只重构终端 UI 层，不重写规则层、存档层和用例语义。

## 目标

- 引入 `rich`，把当前 CLI 升级为面板式终端 UI
- 保持现有中文文案和数字菜单交互
- 明确向 `sts2-cli` 借鉴血条、信息分区、状态摘要和完整决策屏的表现方式
- 保持 `session/use_cases/domain` 的现有边界，不让 UI 逻辑污染规则层

## 非目标

- 不引入 `textual` 或全屏 TUI 框架
- 不修改核心战斗规则、事件规则或存档协议
- 不实现鼠标交互、热键系统或动画
- 不绘制复杂 ASCII 地图
- 不追求与 `sts2-cli` 的状态字段一一对应

## 设计原则

### 一屏一个决策点

每个游戏状态都渲染为一张完整界面，而不是一组零散文本。战斗、事件、奖励、路径选择分别有独立画面，但共用统一的视觉语言。

### 强信息层级

用户最常用、最关键的信息优先展示：

1. 顶部摘要
2. 玩家与敌人状态
3. 手牌或当前主体内容
4. 当前可选操作

### 视觉化状态

尽量把纯文本字段改为更易扫读的终端控件：

- 生命值：血条 + 数值
- 格挡：高亮标签
- 状态：短标签序列
- 敌人意图：图标 + 短文本
- 菜单：编号列表放入固定操作面板

### UI 只读状态

终端 UI 只消费 `SessionState` 和底层 domain model，不直接改写规则状态。所有状态变化仍然由 `session.py` 和 use case 完成。

## 总体架构

终端适配层从单文件字符串渲染改为场景化结构：

```text
session.py
  -> renderer.py
     -> theme.py
     -> widgets.py
     -> screens/
        combat.py
        non_combat.py
        layout.py
```

### 模块职责

#### `theme.py`

统一定义：

- Rich `Theme`
- 面板边框风格
- 颜色 token
- 血条颜色规则
- 常用文本样式名

所有配色和样式都从这里走，避免把颜色散落在各 screen 中。

#### `widgets.py`

提供可复用的终端组件：

- `render_hp_bar(current, maximum)`
- `render_block(value)`
- `render_statuses(statuses)`
- `render_enemy_intent(enemy)`
- `render_menu(options)`
- `render_room_summary(...)`

这些组件只负责局部视图拼装，不做场景路由。

#### `screens/combat.py`

只负责战斗场景，包括：

- 顶部摘要条
- 玩家面板
- 敌方面板
- 手牌面板
- 当前菜单面板

战斗子状态如“选牌”“选目标”只改变底部菜单区，不清空上方上下文。

#### `screens/non_combat.py`

负责非战斗画面：

- 事件
- 奖励
- 路径选择
- 默认房间摘要

重点是统一布局和视觉风格，而不是把所有非战斗类型揉成同一种文本输出。

#### `screens/layout.py`

封装共享布局骨架，例如：

- 顶部摘要 + 中部主体 + 底部操作区
- 双栏内容区
- 标准面板组合方式

这样战斗、事件、奖励画面共用同一套布局骨架。

#### `renderer.py`

仅保留总调度职责：

- 判断当前房间类型
- 判断当前菜单状态
- 选择对应 screen renderer
- 返回终端最终可打印字符串

`renderer.py` 不再包含大段具体排版逻辑。

## 视觉规格

### 借鉴 `sts2-cli` 的点

- 面板化分区
- 颜色区分玩家、敌人、资源、状态
- 使用血条替代单纯数值
- 同屏显示完整决策上下文
- 战斗信息密度高，但仍有清晰层级

### 不直接照抄的点

- 不采用英文命令式交互
- 不显示当前项目尚未实现的大量真实游戏字段
- 不做中英混杂的交互提示

### 终端字符策略

接受并主动使用：

- box drawing 字符：`┌ ┐ └ ┘ │ ─`
- 血条字符：`█ ░`
- 少量图形符号：`⚔ 🛡`

目标是在普通终端中获得接近“控制台 UI”的观感，而不是彩色日志。

## 场景布局

### 战斗屏

战斗屏固定为四段：

#### 1. 顶部摘要条

显示：

- 角色名
- 章节名
- 房间类型
- 回合数
- 当前能量
- 抽牌堆 / 弃牌堆数量

#### 2. 中部双栏

左栏显示玩家：

- 名字
- 血条和数值
- 格挡
- 状态

右栏显示敌人列表：

- 编号
- 名字
- 血条和数值
- 格挡
- 状态
- 意图

#### 3. 下部手牌区

每张牌显示：

- 编号
- 名称
- 费用
- 核心效果摘要

例如：

- `1. 打击 (1) 造成 6 伤害`
- `2. 防御 (1) 获得 5 格挡`
- `3. 重击 (2) 造成 8 伤害，施加 2 层易伤`

#### 4. 底部操作区

常规战斗根菜单显示：

- `1. 查看战场`
- `2. 出牌`
- `3. 结束回合`
- `4. 保存游戏`
- `5. 读取存档`
- `6. 退出游戏`

当菜单模式切换为 `select_card` 或 `select_target` 时，仅替换底部操作区内容，上方战斗信息保持不动。

### 事件屏

布局分三段：

- 顶部摘要条
- 中部事件正文面板
- 底部选项面板

若事件已结算，则中部增加结果面板，用于强调“结果”而不是把结果淹没在正文里。

### 奖励屏

布局分三段：

- 顶部奖励摘要
- 中部奖励列表面板
- 底部操作面板

目标是让“战斗结束 -> 查看奖励 -> 领取奖励”具有独立阶段感。

### 路径选择屏

第一期不绘制整张地图，只渲染当前分支选择：

- 顶部摘要
- 中部可前往房间列表
- 底部编号菜单

避免为了假地图浪费实现成本。

## 交互保持不变

以下行为保持现状：

- 用户仍然只输入数字
- 菜单状态机仍由 `SessionState.menu_state` 驱动
- `route_menu_choice()` 仍然负责输入路由
- `interactive_loop()` 仍然以“渲染 -> 读取输入 -> 路由 -> 再渲染”的循环工作

也就是说，本次改造的是“显示层”，不是“输入协议”。

## 测试策略

### 渲染测试

新增 Rich 渲染相关测试，验证固定状态下的输出包含关键结构：

- 面板标题
- 中文菜单
- 血条字符
- 事件文本
- 奖励项文本

由于终端渲染中包含 Rich 样式控制符，测试应优先断言结构性文本，而不是对整屏 ANSI 输出做脆弱的全量匹配。

### 流程测试

保持和扩展现有 session flow 测试，覆盖：

- 根菜单 -> 选牌 -> 选目标
- 战斗完成 -> 奖励菜单
- 多分支 -> 路径选择
- 存档 -> 读取 -> 正确恢复菜单和屏幕

### 冒烟测试

运行真实 CLI，验证：

- 终端可正常渲染 Rich 输出
- 中文字符不乱码
- 数字菜单可完成一轮基础操作

## 实现顺序

1. 引入 `rich` 依赖和基础主题
2. 拆出 `theme.py`、`widgets.py`、`screens/layout.py`
3. 重做战斗屏
4. 重做事件、奖励、路径选择和默认房间屏
5. 收敛 `renderer.py` 为纯调度层
6. 补渲染测试和 CLI 冒烟测试

## 风险与约束

### ANSI 样式兼容性

不同终端对颜色和框线字符的支持程度不同。第一期默认面向现代终端，不为极老终端做兼容层。

### Rich 输出测试脆弱性

如果直接比较整屏 ANSI 文本，测试会很脆弱。解决方式是：

- 以结构化文本为主断言
- 避免把颜色码当作核心测试目标

### 信息量失衡

参考项目因为跑真实引擎，信息量远大于当前项目。如果强行模仿全部字段，会让画面显得空洞。第一期只渲染当前项目真正拥有的信息。

## 验收标准

- 战斗屏具备面板分区、血条、状态、敌人意图和中文编号菜单
- 事件屏、奖励屏、路径选择屏采用统一的面板化布局
- UI 层改造不改变现有规则语义
- 仍然使用纯中文显示和数字输入
- 终端观感明显从“文本日志”升级为“面板式游戏 CLI”
