# Textual Hover Preview Design

Date: 2026-03-27
Status: Draft for review

## 1. 背景

当前 `Textual` 模式已经具备：

- 左侧地图点击与悬停联动
- 右侧日志、摘要和操作列表
- 奖励与商店沿用统一的 `OptionList` 菜单交互

但奖励和商店列表仍然只有“名称 + 点击领取/购买”这一层信息。用户在领取奖励或浏览商店时，必须先点击、或切换到其他查看路径，才能知道：

- 卡牌具体效果
- 遗物完整效果与副作用
- 药水效果
- Boss 遗物的替换关系与禁用操作

用户本轮目标是：

- 在 `Textual` 模式下，为奖励与商店列表提供**鼠标悬停即显示说明**的预览体验
- 奖励与商店共用一套交互，不分别做两套特判 UI
- 详情显示位置为右侧单独的预览面板，而不是复用顶部摘要

## 2. 目标

- 在 `Textual` 右侧增加统一的“预览详情”区域
- 当鼠标悬停或键盘高亮到奖励/商店物品时，实时显示对应详情
- 覆盖：
  - 战斗奖励
  - Boss 奖励流程中的 Boss 遗物
  - 商店卡牌
  - 商店遗物
  - 商店药水
- 对非物品控制项显示简短操作说明，而不是空白

## 3. 非目标

- 不改动 `route_menu_choice()`、奖励结算、商店购买等领域和用例逻辑
- 不重做 `Textual` 右侧菜单为卡片式图形列表
- 不修改终端版 `rich` 菜单布局
- 不在本轮为所有菜单都引入悬停说明，只覆盖奖励/商店相关模式

## 4. 设计选择

本轮采用：

- **保留现有 `OptionList`，新增共享预览面板**

不采用：

- 为奖励和商店分别写两套悬停说明逻辑
- 自定义一个全新的奖励/商店列表组件替换 `OptionList`
- 用顶部 `action-summary` 替代专门详情区

原因：

- `OptionList` 已具备高亮事件，支持鼠标和键盘共用同一套联动
- 新增独立预览面板最符合用户要求，也不会打乱现有日志区语义
- 共享预览解析层可复用于奖励和商店，后续再扩展其他列表型菜单也容易

## 5. 总体方案

### 5.1 右侧结构调整

在 `src/slay_the_spire/adapters/textual/slay_app.py` 中，右侧区域从：

- 日志
- 操作摘要
- 操作列表
- Flash 信息

调整为：

- 日志
- 操作摘要
- 操作列表
- 预览详情
- Flash 信息

“预览详情”面板只在以下菜单模式下显示有效内容：

- `select_reward`
- `select_boss_reward`
- `select_boss_relic`
- `shop_root`

其他模式下可隐藏，或显示为空占位；本轮优先选“隐藏或清空内容但不报错”的实现。

### 5.2 预览更新触发

预览内容由 `OptionList.OptionHighlighted` 驱动。

这意味着：

- 鼠标移动到某一项时会更新预览
- 键盘上下切换高亮项时也会更新预览
- 点击选中仍沿用现有 `OptionSelected` 提交流程，不影响原有操作

进入奖励/商店菜单后，如还没有高亮项，预览面板先显示引导文案。

### 5.3 共享解析层

在 `Textual` 适配层新增一组纯展示辅助函数：

- 判断当前菜单模式是否支持预览
- 根据当前高亮的 `action_id` 解析预览目标
- 将目标渲染为 Rich 文本或多行字符串

这组函数只消费：

- 当前 `SessionState`
- 当前菜单模式
- 当前高亮项的 `action_id`

不直接驱动会话状态迁移。

## 6. 各菜单模式的预览规则

### 6.1 战斗奖励 `select_reward`

物品项：

- `claim_reward:card_offer:*`：显示卡牌详情
- `claim_reward:card:*`：显示卡牌详情
- `claim_reward:relic:*`：显示遗物详情
- `claim_reward:gold:*`：显示金币奖励说明
- 后续若新增 `claim_reward:potion:*`：显示药水详情

控制项：

- `skip_card_rewards`：显示“跳过当前卡牌奖励，不领取卡牌”
- `claim_all`：显示“领取当前列表中所有可领取奖励”
- `back`：显示“返回上一步”

### 6.2 Boss 奖励主页 `select_boss_reward`

本菜单本身不是物品列表，而是流程入口，因此预览规则为：

- `claim_boss_gold`：显示金币奖励说明
- `claimed_boss_gold`：显示“金币奖励已领取”
- `choose_boss_relic`：显示“进入 Boss 遗物选择”
- `claimed_boss_relic`：显示“Boss 遗物已选择”
- `back`：显示“返回上一步”

### 6.3 Boss 遗物选择 `select_boss_relic`

物品项：

- `claim_boss_relic:*`：显示遗物详情

控制项：

- `back`：显示“返回上一步”

### 6.4 商店 `shop_root`

物品项：

- `buy_card:*`：显示卡牌详情
- `buy_relic:*`：显示遗物详情
- `buy_potion:*`：显示药水详情

控制项：

- `remove`：显示删牌服务说明
- `leave`：显示“离开商店”
- `inspect`：显示“查看资料”
- `save`：显示“保存游戏”
- `load`：显示“读取存档”
- `quit`：显示“退出游戏”

## 7. 详情内容设计

### 7.1 卡牌

优先复用现有卡牌详情/摘要能力，至少显示：

- 名称
- 费用
- 类型
- 效果摘要
- 若存在，升级目标

奖励和商店中的卡牌预览应尽量复用已有的卡牌说明逻辑，而不是重新手写一套文本。

### 7.2 遗物

优先复用现有遗物详情能力，至少显示：

- 名称
- 主要效果
- 摘要/描述
- 替换原遗物（如 `black_blood`）
- 禁用操作
- 金币规则
- 触发 Hook 概览

### 7.3 药水

当前项目尚无完整的药水详情面板辅助函数，但内容模型足够简单，可新增轻量格式化函数，至少显示：

- 名称
- 药水 ID
- 效果摘要

药水预览风格应与卡牌、遗物保持一致，但不需要在本轮引入新的 inspect 菜单。

### 7.4 控制项

非物品项统一显示：

- 一行标题
- 一到两行操作说明

避免出现：

- 空白预览
- 伪造物品详情
- 用原始 action id 直接暴露英文标识

## 8. UI 行为细节

### 8.1 默认态

进入支持预览的菜单但尚未高亮具体项时，预览面板显示引导文案，例如：

- “将鼠标移到奖励上查看说明”
- “将鼠标移到商品上查看说明”

### 8.2 键鼠一致

预览更新必须同时支持：

- 鼠标悬停
- 键盘高亮移动

不能只支持鼠标，否则会让键盘用户丢失信息。

### 8.3 菜单切换清理

离开支持预览的菜单后：

- 预览面板应清空或隐藏
- 不能把旧奖励/商店说明残留到其他界面

### 8.4 未知项兜底

如果某个 action id 已进入支持预览的菜单，但当前没有专门格式化逻辑：

- 预览面板显示“暂未提供详细说明”
- 可附带当前中文标签

不能抛异常或导致界面更新失败。

## 9. 实施范围

主要修改：

- `src/slay_the_spire/adapters/textual/slay_app.py`
- `src/slay_the_spire/adapters/terminal/inspect.py`

可能需要少量辅助读取：

- `src/slay_the_spire/app/menu_definitions.py`

其中：

- `slay_app.py` 负责右侧布局、事件订阅、预览面板刷新和 menu mode 分发
- `inspect.py` 负责复用或补齐卡牌/遗物/药水的展示文本格式化

## 10. 测试计划

新增或补充 `tests/adapters/textual/test_slay_app.py`，至少覆盖：

1. 奖励菜单下会出现预览面板
2. 高亮卡牌奖励时显示卡名和效果摘要
3. 高亮 Boss 遗物时显示遗物名和效果摘要
4. 高亮商店卡牌时显示卡牌说明
5. 高亮商店遗物时显示遗物说明
6. 高亮商店药水时显示药水说明
7. 高亮 `claim_all`、`back`、`remove` 等控制项时显示操作说明
8. 切出奖励/商店菜单后，旧预览不会残留

## 11. 风险与取舍

- `OptionList` 的高亮状态与鼠标行为依赖 `Textual` 组件机制；测试需覆盖鼠标和默认高亮行为，避免只在人工操作下可见
- 奖励和商店的 action id 结构不同，预览解析必须集中管理，否则容易分散在多个 if 分支里变脆
- 药水目前缺少现成详情格式化函数，本轮新增时应保持最小实现，不借机扩张 inspect 系统

## 12. 结论

本轮以“共享悬停预览层”为核心：

- 不改现有菜单提交逻辑
- 不重写 `OptionList`
- 在 `Textual` 右侧增加独立预览面板
- 用统一解析层同时覆盖奖励与商店

这样可以用最小的结构改动，为 `Textual` 模式补上更接近游戏体验的信息预览能力。
