# Textual Map Original-Style Design

Date: 2026-03-27
Status: Draft for review

## 1. 背景

项目已经引入 `Textual` 地图视图，具备：

- 左侧可滚动地图
- 节点点击选择
- 右侧日志与可点击菜单
- 当前节点自动定位

但当前地图视觉语言仍然明显偏“程序生成 TUI”：

- 节点是大块彩色方框
- 连线是直角流程图式走线
- 节点布局偏规则网格
- 地图整体更像控制台拓扑图，不像《Slay the Spire》原版地图

用户本轮目标不是重写地图规则，而是让 `Textual` 模式下的地图**学原版显示方式**，重点靠近原版的：

- 自下往上的阅读方向
- 稀疏留白的路径排布
- 图标化节点
- 更接近手绘感的虚线连线

同时用户明确要求：

- **保留房间颜色语义**
- **节点显示为“图标 + 小字”**
- **地图区域可以改成浅色纸面背景**
- **右侧信息区继续保持当前深色 UI**

## 2. 目标

- 将 `Textual` 地图从“方框流程图”改为“原版风格地图”
- 保持当前状态机、编号菜单和点击交互不变
- 让地图主视图改为**自下往上推进**
- 保留房间颜色提示，但弱化为节点级提示，而不是整块色块
- 保留滚动、自动定位、可点击操作与右侧菜单联动

## 3. 非目标

- 不重写领域层地图生成逻辑
- 不修改 `ActState` / `ActNodeState` 数据结构
- 不实现原版像素级材质、动画或完整贴图复刻
- 不把右侧菜单改造成完全图形化界面
- 不在本轮追求原版地图生成规则的高拟真复刻

## 4. 设计选择

本轮采用：

- **UI 层重绘的原版式地图**

不采用：

- 只替换节点图标、不调整布局与连线
- 直接重写地图生成算法去复刻原版路径分布
- 继续使用大方框节点，只在配色上做微调

原因：

- 用户要的是“学原版的显示方式”，核心在**视觉语言和路径阅读方式**
- 当前 `ActState` 已经足够支撑一层展示坐标变换，无需先动领域层
- 仅改 UI 层可以把风险集中在 `Textual` 适配层，避免影响终端版和地图规则

## 5. 总体方案

### 5.1 界面分层

- 左侧地图区改成**浅色纸面背景**
- 右侧日志/菜单区继续使用深色面板
- 左侧承担“空间导航”和“路径感知”
- 右侧承担“详细文字说明”和“可点击操作”

这种分层允许地图更接近原版视觉，同时不牺牲现有信息可读性。

### 5.2 阅读方向

- 地图主视图改成**自下往上阅读**
- 底部代表起点/已走层
- 顶部代表未来路径与 Boss
- 初始定位落在当前节点附近，并让视口上方预留主要观察空间

这会让地图浏览方式更接近原版“向上爬塔”的感受。

### 5.3 节点风格

节点从当前的大方框改为：

- 一个主图标
- 图标下方一行小字标签

例如：

- `⚔ 战斗`
- `☠ 精英`
- `👑 Boss`
- `? 事件`
- `🛒 商店`
- `🔥 休息`

节点面积应明显小于现状，让地图拥有更多留白。

### 5.4 颜色策略

保留颜色语义，但收敛使用范围：

- 颜色作用于节点本身、描边和状态高亮
- 不再使用整块大面积底色方框

状态表达建议：

- 当前节点：最强高亮（亮描边/光圈）
- 可达节点：次级高亮（明显可点击）
- 不可达节点：降亮度、降饱和

如果现有状态难以稳定区分“已走过节点”，本轮不强行新增该状态。

## 6. 布局与渲染设计

### 6.1 保持领域模型不变

继续使用现有：

- `ActNodeState.row`
- `ActNodeState.col`
- `ActNodeState.next_node_ids`

不在本轮修改领域层地图结构。

### 6.2 引入展示坐标层

在 `MapWidget` 内新增“展示坐标”计算，而不是直接用：

- `x = col * fixed_spacing`
- `y = row * fixed_spacing`

展示坐标规则：

- 纵向：严格按 `row` 自下往上排列
- 横向：以 `col` 为基础列，再叠加轻微偏移

目标是形成：

- 仍可控、仍稳定
- 但不完全规则对齐
- 更像原版的自然分叉感

偏移必须是稳定的，不能因为重绘或滚动而变化。

### 6.3 节点命中区域

节点视觉体积变小后，仍需保留足够可点击面积：

- 命中区域覆盖图标与小字整体
- 命中区域可略大于可见像素范围
- 鼠标悬停与点击继续沿用当前节点区域命中机制

### 6.4 连线策略

连线从“直角流程图”改为“原版风格的稀疏虚线路径”。

终端里不追求真正贝塞尔曲线，而采用：

- 分段斜向路径
- 适度稀疏的 dash 模式
- 避免大段横平竖直直角折线

目标不是几何精确，而是摆脱流程图感，形成“手绘路线”观感。

### 6.5 保留单组件自绘

继续以单个自绘 `ScrollView` 承载整张地图，而不是拆成大量小组件。

原因：

- 当前已经验证了 `ScrollView + virtual_size` 的滚动模型可用
- 地图需要统一处理滚动、悬停、点击、自动定位
- 单组件自绘更容易控制整图布局、虚线连线和命中区域

## 7. 交互与右侧联动

### 7.1 地图交互

- 当前节点不可点击
- 可达节点可点击
- 不可达节点只提供浏览，不触发选择
- 悬停时反馈应轻量，不破坏整体地图观感

### 7.2 右侧联动

右侧继续保留：

- 游戏日志
- 可点击操作列表
- 状态提示

地图本身不承载过多说明文字。  
如需补充提示，建议仅在右侧顶部增加轻量悬停信息，例如：

- `当前悬停：精英房（可达）`
- `当前悬停：商店（不可达）`

### 7.3 滚动与初始视口

- 保留真实滚动条
- 初始打开时定位到当前节点附近
- 视口应略偏向“当前节点以下一点、以上更多”，方便用户向上看未来路径
- 进入新节点后重新定位，但避免过于突兀的动画

## 8. 实施边界

本轮主要修改：

- `src/slay_the_spire/adapters/textual/map_widget.py`
- 少量联动代码位于 `src/slay_the_spire/adapters/textual/slay_app.py`
- 对应测试位于 `tests/adapters/textual/test_slay_app.py`

明确不改：

- `domain/` 地图生成逻辑
- 终端版 `rich` 地图表现
- `route_menu_choice()` 与现有菜单状态机

## 9. 测试策略

重点验证 `Textual` 适配层，而不是领域层。

建议测试点：

- 地图仍有真实滚动范围
- 初始视口定位在当前层附近
- 节点点击仍能触发正确选择
- 右侧菜单不重复渲染
- 当前节点/可达节点在新风格下仍能正确区分
- 地图渲染不因浅色背景与新样式导致样式异常

如果现有测试更偏结构断言，本轮允许更多校验：

- 展示尺寸大于视口
- 当前节点位置参与滚动定位
- 节点区域映射存在且可点击

## 10. 风险与取舍

### 10.1 风险

- 终端字符画难以精确模拟原版曲线
- 节点体积变小后，点击热区若过窄会降低可用性
- 浅色地图与深色右栏共存，需要避免视觉割裂

### 10.2 取舍

本轮优先级是：

1. 原版式阅读方式
2. 节点与连线视觉语言
3. 保持交互稳定

不追求：

- 原版贴图级视觉拟真
- 原版地图生成算法拟真

## 11. 结论

本轮将 `Textual` 地图重绘为：

- 浅色纸面背景
- 自下往上阅读
- 图标 + 小字节点
- 稀疏虚线连线
- 节点级颜色提示
- 当前/可达状态强化高亮

同时保持：

- 当前状态机
- 右侧日志与可点击菜单
- 滚动与自动定位
- 鼠标点击选点

这样可以在不重写地图规则的前提下，把地图显示方式明显拉近原版《Slay the Spire》。
