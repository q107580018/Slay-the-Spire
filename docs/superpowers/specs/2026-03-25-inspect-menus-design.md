# Inspect Menus and Detail Panels Design

Date: 2026-03-25
Status: Draft for review

## 1. 背景

当前项目已经有可玩的编号菜单主流程，但“信息检查”能力仍停留在摘要层：

- 战斗中只能看到简化版手牌和敌人摘要
- 非战斗中只能看到房间摘要与地图摘要
- 玩家无法像原版《杀戮尖塔》那样随时检查长期资源
- 玩家也无法像 `sts2-cli` 那样随时展开牌组、遗物、药水和更完整的说明

这直接影响可玩性。对《杀戮尖塔》这类牌组构筑游戏来说，决策质量高度依赖“随时检查当前拥有的资源、对象效果和战斗信息”。

## 2. 参考锚点

本设计参考两个方向：

- 原版《Slay the Spire》的检查习惯
- [`wuhao21/sts2-cli`](https://github.com/wuhao21/sts2-cli) 的终端信息组织

明确参考点：

- `sts2-cli` 在任意提示下暴露 `deck / relics / potions / map` 这类随时检查能力
- `sts2-cli` 不只列名字，也会把遗物、药水、整副牌的描述展开出来
- 原版 STS 的核心体验不是“所有内容都常驻屏幕”，而是“玩家能在需要时低成本查看详细信息”

本项目不会照搬 `sts2-cli` 的纯 ANSI 文本外观，而是保留现有 `Rich` 面板式界面，吸收它的信息架构优点。

## 3. 目标

- 在所有主要场景中提供统一的“查看资料”入口
- 支持查看 `属性 / 牌组 / 遗物 / 药水`
- 在战斗中额外支持查看 `手牌 / 抽牌堆 / 弃牌堆 / 消耗堆 / 敌人详情`
- 支持从列表进入单对象详情，查看完整效果说明
- 让检查页视觉上比当前摘要更完整、更整洁，但不挤爆主界面
- 保持现有“编号菜单驱动”交互，不引入自由文本命令模式

## 4. 非目标

- 不改为 `sts2-cli` 那种 `deck` / `relics` 文本命令交互
- 不实现搜索、排序、筛选、分页切换等高级浏览器式能力
- 不实现鼠标、悬浮、键盘方向键或多快捷键体系
- 不在第一轮实现敌人未来多轮意图预测
- 不重做整体主题，仅在当前 `Rich` 结构上增强检查页表现

## 5. 设计原则

### 5.1 检查入口统一

玩家不应记忆每个房间独有的一套查看命令。所有主要场景都提供同一个入口：

- `查看资料`

进入后再按上下文展开可检查的内容。

### 5.2 摘要页保持紧凑，详情页负责完整信息

主界面继续承担“快速扫一眼”的职责：

- 战斗摘要
- 房间摘要
- 当前手牌简表
- 当前敌人简表

完整信息放到 inspect 页处理，避免主画面过载。

### 5.3 编号菜单优先，不引入文本命令

当前项目的核心交互已经是 `route_menu_choice()` 的编号菜单。本设计继续沿用：

- 根菜单进入检查菜单
- 检查菜单进入列表
- 列表再进入详情

这样最符合当前架构，也更适合后续补终端测试。

### 5.4 视觉上要“像一个检查面板”，不是把日志堆成一坨

检查页不应只是简单输出多行字符串。需要利用现有 `Rich Panel` 和布局能力，做到：

- 标题明确
- 重点字段有视觉分组
- 长期资源与战斗资源分区
- 列表与详情有明显层次

## 6. 用户体验

### 6.1 根菜单新增入口

所有主菜单新增一项 `查看资料`。

受影响场景包括：

- 战斗未结束
- 非战斗普通房间
- 事件房间
- 战斗后奖励房间
- 商店
- 休息点
- 终局画面

编号会随场景调整，但“查看资料”在每个根菜单都存在。

### 6.2 Inspect 根菜单

进入 `查看资料` 后，先看到“资料总览”菜单。

始终可见项：

- `属性`
- `牌组`
- `遗物`
- `药水`

战斗中额外显示：

- `手牌`
- `抽牌堆`
- `弃牌堆`
- `消耗堆`
- `敌人详情`

最后一项固定为：

- `返回上一步`

### 6.3 列表页与详情页

`牌组 / 手牌 / 抽牌堆 / 弃牌堆 / 消耗堆`

- 列表页显示卡牌名、费用、类型、简短效果摘要
- 选择某张牌后进入卡牌详情页

`遗物 / 药水`

- 列表页显示名称和一行简短说明
- 选择某个条目后进入详情页

`敌人详情`

- 列表页显示敌人名、生命、格挡、状态、当前意图
- 选择某个敌人后进入详情页

`属性`

- 直接进入详情面板，无需二级列表

## 7. 详情内容规范

### 7.1 属性详情

属性详情至少显示：

- 角色名
- 当前生命 / 最大生命
- 当前金币
- 当前章节
- 当前房间
- 当前房间类型
- 当前牌组数量
- 当前遗物数量
- 当前药水数量
- 当前删牌次数

战斗中额外显示：

- 当前回合
- 当前能量
- 当前格挡
- 玩家状态
- 抽牌堆数量
- 弃牌堆数量
- 消耗堆数量

### 7.2 卡牌详情

卡牌详情至少显示：

- 卡牌名
- 实例 ID
- 费用
- 类型
- 是否可打出
- 完整效果说明
- 升级目标

如果卡牌已升级：

- 名称中体现升级态
- 详情里显示“已升级”

如果未来要支持关键词，本层就是展示入口。第一轮可先用已有效果和升级字段。

### 7.3 遗物详情

遗物详情至少显示：

- 遗物名
- 完整效果说明
- 触发时机摘要

由于当前数据模型里遗物定义是：

- `trigger_hooks`
- `passive_effects`

第一轮可用人类可读的效果格式化层把它们转成中文说明，而不是直接把原始字段名暴露给玩家。

### 7.4 药水详情

药水详情至少显示：

- 药水名
- 完整效果说明

如果未来支持药水目标类型，这里也是自然展示位。

### 7.5 敌人详情

敌人详情至少显示：

- 敌人名
- 当前生命 / 最大生命
- 当前格挡
- 当前状态
- 当前意图摘要
- 招式表预览

第一轮的“招式表预览”可以是保守版本：

- 当前意图
- 已知 move table 的简要轮播信息

不承诺等同于原版的完整 AI 预判界面。

## 8. 渲染方案

## 8.1 总体策略

不重做应用壳层，继续复用：

- `build_standard_screen()`
- `build_two_column_body()`
- `render_menu()`

检查页的核心是让正文区域切换成 inspect 内容。

### 8.2 战斗中的检查页

战斗 inspect 视图仍保留顶部战斗摘要。

正文区域建议改为：

- 左列：玩家或对象概览
- 右列：选中对象的详情，或当前列表说明

如果当前是列表页：

- 左侧列表
- 右侧说明面板

如果当前是详情页：

- 左侧对象摘要
- 右侧完整详情

底部仍是编号菜单。

### 8.3 非战斗中的检查页

非战斗 inspect 视图仍保留顶部房间摘要。

正文区域建议改为：

- 上方或左侧保留地图/房间简图
- 另一侧显示 inspect 面板

重点是避免完整地图把详情页挤碎。第一轮允许在 inspect 模式下弱化地图存在感，例如：

- 保留一块较窄地图面板
- 把 inspect 主内容放到更大区域

### 8.4 视觉规范

检查页要比当前更“像游戏内情报页”。推荐做法：

- 列表面板标题明确，例如 `牌组列表`、`遗物列表`
- 详情面板标题带对象名，例如 `卡牌详情：重击`
- 关键字段使用 `summary.label` 一类样式前缀
- 卡牌效果、遗物效果使用换行分段，而不是全挤成一行
- 战斗堆区数量在标题或副标题中直接显示

## 9. 状态机设计

现有 `MenuState` 只有：

- `mode`
- `selected_card_instance_id`

本次需要扩展 inspect 导航上下文。

建议 `MenuState` 增加通用字段：

- `mode`
- `selected_card_instance_id`
- `inspect_section`
- `inspect_item_id`

或采用更直接的模式枚举：

- `inspect_root`
- `inspect_stats`
- `inspect_deck`
- `inspect_card_detail`
- `inspect_relics`
- `inspect_relic_detail`
- `inspect_potions`
- `inspect_potion_detail`
- `inspect_hand`
- `inspect_draw_pile`
- `inspect_discard_pile`
- `inspect_exhaust_pile`
- `inspect_enemies`
- `inspect_enemy_detail`

本次推荐“显式 mode”方案。原因：

- 现有 `session.py` 就是按 `mode` 分发
- 终端渲染层也是按 `mode` 决定 footer
- 测试时更容易断言当前状态

### 9.1 详情页返回策略

详情页返回应回到来源列表，而不是直接回根菜单。

例如：

- 从 `inspect_deck` 进入 `inspect_card_detail`
- 返回后回到 `inspect_deck`

因此需要在 `MenuState` 中保存详情来源列表模式，避免回退错位。

## 10. Session 路由设计

`route_menu_choice()` 需要新增 inspect 相关分支。

建议增加：

- `_route_inspect_root_menu()`
- `_route_inspect_card_list_menu()`
- `_route_inspect_relic_list_menu()`
- `_route_inspect_potion_list_menu()`
- `_route_inspect_enemy_list_menu()`
- `_route_inspect_detail_menu()`

根菜单逻辑只负责：

- 进入 inspect 根菜单

inspect 子路由负责：

- 切换 section
- 选择对象
- 返回上一层

这样可以避免 `_route_root_menu()` 进一步膨胀。

## 11. 数据格式化层

目前已有：

- `summarize_effect()`
- `summarize_card_effects()`
- `preview_enemy_intent()`

但这还不够支撑完整详情页。

建议补一组 inspect 专用格式化函数，放在终端层，例如：

- `format_card_detail_lines()`
- `format_relic_detail_lines()`
- `format_potion_detail_lines()`
- `format_enemy_detail_lines()`
- `format_run_stats_lines()`

理由：

- 这些是展示逻辑，不应放进 domain
- 现有 content 定义提供的是结构化数据，不是最终玩家文案
- 终端层可统一把效果转成中文

## 12. 与当前内容模型的兼容性

本次不要求改动存档 schema 或运行态结构。

已经可直接复用的数据包括：

- `RunState.deck`
- `RunState.relics`
- `RunState.potions`
- `RunState.gold`
- `RunState.current_hp`
- `RunState.max_hp`
- `RunState.card_removal_count`
- `CombatState.hand`
- `CombatState.draw_pile`
- `CombatState.discard_pile`
- `CombatState.exhaust_pile`
- `CombatState.player`
- `CombatState.enemies`

如果 `CombatState` 当前没有 `exhaust_pile`，则本次实现需要先核实实际字段；若尚不存在，可用空列表兜底，但 UI 仍保留该入口。

## 13. 测试策略

### 13.1 终端层测试

补充 `tests/adapters/terminal/`：

- inspect 根菜单渲染
- 牌组列表渲染
- 卡牌详情渲染
- 遗物详情渲染
- 属性详情渲染
- 战斗 inspect 菜单渲染

### 13.2 Session 路由测试

补充 `tests/app/` 或现有合适位置：

- 从战斗根菜单进入 inspect 根菜单
- 从非战斗根菜单进入 inspect 根菜单
- 从列表进入详情再返回来源列表
- inspect 模式下保存/读取/退出的编号仍正确

### 13.3 E2E 冒烟测试

更新 `tests/e2e/test_single_act_smoke.py`：

- 在至少一个战斗场景进入 inspect
- 在至少一个非战斗场景进入 inspect
- 确认 inspect 流程不会破坏原有主路径推进

## 14. 实现顺序

建议按以下顺序实施：

1. 补设计所需的 `MenuState` / inspect mode 路由
2. 先实现非战斗 inspect：
   - `属性`
   - `牌组`
   - `遗物`
   - `药水`
3. 再实现战斗 inspect：
   - `手牌`
   - `抽牌堆`
   - `弃牌堆`
   - `消耗堆`
   - `敌人详情`
4. 最后统一打磨视觉和菜单编号

## 15. 风险与缓解

### 15.1 菜单编号漂移

各场景根菜单项数不同，新增 `查看资料` 后容易导致测试和用户记忆错位。

缓解：

- inspect 入口在各根菜单中的位置尽量保持稳定
- 同步更新 smoke test

### 15.2 `session.py` 路由继续膨胀

如果继续把所有分支都堆在 `_route_root_menu()`，后续会更难维护。

缓解：

- inspect 路由单独拆辅助函数

### 15.3 详情文案不够人类可读

当前数据模型偏结构化，若直接输出字段名会显得很差。

缓解：

- 增加终端专用中文格式化层
- 对未知 effect type 提供保守降级文案，而不是暴露原始 JSON

## 16. 推荐结论

本次推荐实现“统一检查入口 + inspect 列表页 + inspect 详情页”的分层方案：

- 既保留原版 STS 的随时检查习惯
- 又吸收 `sts2-cli` 的信息完备性
- 同时继续贴合当前项目的 `Rich + 编号菜单` 架构

最终用户感知应为：

- 平时看紧凑摘要
- 想看清楚时随时点开资料页
- 卡牌、遗物、药水、战斗堆区和敌人都能查到明确效果说明
